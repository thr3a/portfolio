# 目的

ユーザーが新機能や新規プロジェクトをふんわりした粒度で依頼してきたとき、いきなり実装に入らず、**AskUserQuestion ツールを使って質問を投げ、要件を確定させてから着手する**ためのスキル。

「あとから手戻りが発生する」「ユーザーの意図と違うものを作ってしまう」のを防ぐのが狙い。

# 手順

## 1. 依頼の解釈

まず依頼文をそのまま受け取り、何を作ろうとしているのかを 1〜2 文で言語化する。
このとき、自分の中で「決め打ちで進められる部分」と「ユーザーに確認しないと決められない部分」を分ける。

## 2. 懸念点・確認事項の洗い出し

以下の観点で確認すべき項目を洗い出す。網羅的にやる必要はなく、**そのプロジェクトの本質に関わる順**に並べる。

- **入力**: 何をどう受け取るのか（形式、サイズ上限、文字コード、複数対応の有無）
- **出力**: 何をどう返すのか（形式、サイズ、命名、保存先、ダウンロード方法）
- **処理ロジック**: 変換ルール、エッジケースの扱い、エラー時の挙動
- **UI/UX**: 画面構成、操作フロー、レスポンシブ対応、ダーク/ライトモード
- **技術スタック**: フレームワーク、ライブラリ、ホスティング、サーバ有無
- **非機能要件**: 認証、課金、レート制限、保存期間、プライバシー
- **スコープ**: MVP の範囲、将来拡張する予定の機能

## 3. AskUserQuestion で質問する

**必ず Claude Code の AskUserQuestion ツールを使う**。テキストで「〜はどうしますか？」と聞くだけで済ませない。

質問は以下のルールで設計する。

- 1 回の AskUserQuestion で最大5問まで。優先度の高いものから選ぶ
- 各質問の選択肢は **2〜4 個** に絞る。「Other」は自動で追加されるので含めない
- 選択肢には `description` を必ず付け、トレードオフや影響範囲を説明する
- 推奨案がある場合は先頭に置き、ラベル末尾に `(推奨)` を付ける
- `header` は 12 文字以内の短いタグ（例: "入力形式", "技術スタック", "認証"）
- 自明な質問・どうでもいい質問はしない。**作る/作らないの方向性が変わる質問** を優先

質問項目が5つを超える場合は、最初の回答を受けてから次の AskUserQuestion を発動する（段階的に詰める）。

## 4. 回答を受けてサマリを返す

回答が揃ったら、確定した仕様を箇条書きで 1 回まとめて提示する。
そのうえで「この内容で実装に入っていいか」を最後に確認してから着手する。

# 注意

- AskUserQuestion を 絶対にスキップしない。テキストの箇条書きで「以下どうしますか？」と聞くだけでは不十分
- 質問は「ユーザーがその場で即答できる粒度」に分解する。曖昧な質問は再質問を生むのでコスト高
- 一度の AskUserQuestion で全部聞こうとしない。本質的な3〜5問に絞り、必要なら段階的に追加質問する
